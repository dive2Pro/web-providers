### 讲稿

今天我想分享的是当前这个项目的整体架构。这个项目的核心目标，不是直接封装官方 API，而是把 DeepSeek Web 的能力桥接成一个本地可调用的 provider。这样上层运行时在使用它的时候，可以尽量按照 provider 的方式来调用，而不需要直接感知网页侧的细节。

今天我主要讲三件事：第一，这个项目为什么要这么做；第二，这套架构是怎么分层的；第三，一条请求是怎么从上层一路走到网页，再把结果带回来的。最后我会用几分钟补充一下，它和真实 API 调用的不同，以及这套方案本身的限制。

## 第 2 页

### 标题

这个项目要解决什么问题

### 上屏文案

- 上层需要一个统一的 provider 接口
- DeepSeek Web 不是标准服务端 API
- 目标是把 Web 能力转成稳定可用的本地能力

### 配图建议

左右对比图：

- 左边：上层想要的 provider 调用
- 右边：底层实际面对的网页会话和页面事件

### 讲稿

先讲问题本身。上层运行时希望面对的是一个统一的 provider 接口，也就是给它消息、上下文、工具定义，然后拿回结构化结果。

但 DeepSeek Web 本身并不是一个为我们当前接入方式准备好的标准服务端 API。它更像是一个网页产品，里面有页面状态、有会话、有事件流、有交互过程。

所以这个项目做的事情，本质上就是在中间补一层桥接。上层继续按照 provider 的方式使用能力，底层由我们去处理网页绑定、自动化操作、事件解析、结果转译这些复杂度。也就是说，我们不是在直接调用模型服务，而是在把网页侧能力转成一个本地可复用的 provider。

### 标题

一张图看整体架构

### 上屏文案

- `Extension Runtime`
- `Helper(Fastify)`
- `Browser Client + Page Bridge + DeepSeek Web`

### 配图建议

正式分层图，从左到右：

- `Extension Runtime`
- `Helper / Fastify`
- `Browser Client`
- `Page Bridge`
- `DeepSeek Web`

每层下面只写一句职责说明。

### 讲稿

整个系统可以分成几层来看。最上层是 Extension Runtime，它负责对接上层调用语义，也就是 provider 的输入输出、消息组织、工具能力这些。

中间是本地 Helper，它是一个 Fastify 服务。它负责鉴权、路由、状态管理、请求调度，以及把 extension 发来的请求转交给浏览器侧。

再往下是 Browser Client 和 Page Bridge。Browser Client 负责浏览器自动化动作，比如绑定 tab、发 prompt、开始新会话。Page Bridge 负责真正接住网页里的事件和页面状态，把网页里发生的事情转成我们能理解的结构化结果。最底层才是 DeepSeek Web 本身。

所以你可以把这套架构理解成：上层是 provider 语义，中间是本地服务边界，底层是网页能力采集和协议重建。

## 第 4 页

### 标题

为什么要分成这几层

### 上屏文案

- 运行时语义和网页自动化语义不是一回事
- Helper 让状态和接口边界更清晰
- Bridge 负责吸收网页侧不稳定性

### 配图建议

职责分离图：

- 上层关注：`messages / tools / thinking`
- 下层关注：`tab / DOM / SSE / page state`
- 中间放 `Helper + Bridge` 作为隔离层

### 讲稿

这里最重要的问题其实不是能不能跑起来，而是为什么不把这些逻辑都堆在一起。

原因在于，运行时的语义和网页自动化的语义完全不是一回事。上层关心的是 messages、tool calls、thinking、finish reason；浏览器侧关心的是 tab、页面状态、SSE 片段、页面是否 ready。

如果把这两类逻辑直接揉在一起，代码会很快变得不可维护。所以中间需要一个 Helper，把本地 HTTP 边界、状态管理和调度逻辑单独收敛起来。

再往下，Bridge 的作用也很关键。网页侧本来就比标准 API 更不稳定，所以我们把页面解析和事件重建单独隔离出来，让上面几层尽量看到的是稳定 contract，而不是页面细节。

## 第 5 页

### 标题

一次请求是怎么走完整条链路的

### 上屏文案

- Runtime 组织请求
- Helper 找到绑定会话并发起调用
- Browser / Bridge 提交 prompt 并回传结果

### 配图建议

时序图，参与方：

- User / Runtime
- Helper
- Browser Client
- Page Bridge
- DeepSeek Web

### 讲稿

接下来我用一次请求把整个系统串起来。

第一步，上层调用 provider，Runtime 会把当前消息上下文整理成请求。这里会带上 model、messages，必要时也会带 session 初始化信息、工具定义、一些控制参数。

第二步，请求进入 Helper。Helper 会先做几个关键判断，比如当前有没有绑定 DeepSeek tab，当前是不是已经有请求在跑，是否需要开启一个 fresh session。它还会把这次请求登记到本地状态里，便于后续调试和恢复。

第三步，Helper 调 Browser Client 去真正和网页交互。Browser Client 负责把 prompt 发到页面上，Page Bridge 负责监听页面里的 completion 过程，把 streaming 过程里的内容一点点收集回来。

最后，Bridge 把结果整理成统一结构，交回 Helper，Helper 再返回给 Runtime，Runtime 再把它映射成上层能消费的 assistant message 内容。

## 第 6 页

### 标题

Helper 为什么要单独存在

### 上屏文案

- 本地鉴权和路由收口
- 状态集中管理
- 浏览器控制与上层调用解耦

### 配图建议

一个收口层示意图：

- 左边多个入口：`bind / health / reset / provider chat / debug`
- 中间：`Helper State + Routes`
- 右边：`Browser Client`

### 讲稿

单独拉出一个 Helper，是这套架构里一个很关键的工程决策。

第一，它把本地能力变成了明确的服务边界。比如 `bind`、`health`、`reset`、`provider chat` 这些动作，都有统一入口，而不是散落在 extension 逻辑里。

第二，它把状态集中起来了。当前项目里，至少有几类状态是必须集中维护的，比如当前绑定的 session、当前是否有 active request、最近一次 provider request 的调试记录、bridge 的心跳状态、系统是否 degraded。

第三，它把浏览器控制和上层调用解耦了。这样以后无论是换浏览器接入方式，还是补充 debug 能力，改动都更容易收敛在 Helper 这一层，而不是影响整条调用链。

## 第 7 页

### 标题

Page Bridge 真正在做什么

### 上屏文案

- 监听页面 completion 过程
- 从片段事件里恢复完整结果
- 把网页行为转成结构化输出

### 配图建议

碎片到结构的图：

- 左边：`THINK / RESPONSE / partial fragments / completion events`
- 中间：`Page Bridge Parser`
- 右边：`thinkingText / outputText / toolCall?`

### 讲稿

很多人第一次看这个项目，会以为底层只是自动点开网页，然后抓一下文本。但实际难点不在这里。

真正困难的地方在于，网页里的返回不是天然就是一个完整、稳定、结构化的 API 响应。页面侧可能是事件流、SSE 片段、局部更新，里面还可能混有不同类型的内容。

所以 Page Bridge 的职责，其实是做协议重建。它要知道当前 completion 有没有开始、有没有结束、哪些片段属于 thinking、哪些属于最终 response、有没有可能包含 tool call。最后它再把这些碎片重新拼成一份比较像 provider contract 的结果。

换句话说，这一层不是页面抓取，而是网页协议转译。

## 第 8 页

### 标题

为什么结果要统一成 text / tool_call / thinking

### 上屏文案

- `text`：最终用户可见回答
- `tool_call`：结构化工具调用
- `thinking`：推理过程，与 answer 分离

### 配图建议

三个并列卡片，下方一条汇总箭头：

`统一 contract -> 上层统一消费`

### 讲稿

如果只返回一段纯文本，这个项目其实会简单很多。但问题是，上层真正需要的结果并不只是文本。

第一类当然是 `text`，也就是最终回答。

第二类是 `tool_call`。对上层运行时来说，工具调用不是一段普通文本，而是一种需要被执行的结构化意图。所以我们必须把它识别出来，而不是把它埋在文本里。当前实现里，这部分既有 native 检测，也有 JSON fallback。

第三类是 `thinking`。最近这个项目里有一个很重要的演进，就是把思考过程和最终回答分开。因为如果不分开，thinking 内容可能会混进最终回复，既影响渲染，也影响上层消费语义。

所以这三种返回形态，其实定义了这套系统对外暴露的核心 contract。

## 第 9 页

### 标题

三个关键实现点

### 上屏文案

- `session bind`
- `tool call`
- `thinking separation`

### 配图建议

三列布局：

- `Session Bind`：请求落到指定 tab
- `Tool Call`：native detect -> JSON fallback
- `Thinking Separation`：THINK 流和 RESPONSE 流分离

### 讲稿

如果要从实现细节里挑三个最值得讲的点，我会选这三个。

第一是 `session bind`。这解决的是请求到底落到哪个网页会话上的问题。因为这里不是直接调云端 API，而是依赖一个实际存在的 tab，所以会话绑定是整个系统能否稳定工作的前提。

第二是 `tool call`。网页侧拿到的内容未必天然就是标准 tool call 协议，所以我们做了两层策略：能检测到原生结构时走 native；检测不到时，再尝试从严格 JSON 协议里做 fallback 修复。

第三是 `thinking separation`。这背后不是简单加个字段，而是底层 bridge 必须识别不同 fragment 类型，并在 Runtime 层把 thinking 和 text 映射成不同内容项。这个改动让上层 finally 能把 reasoning 和 final answer 分开处理。

## 第 10 页

### 标题

它和真实 API 调用有什么不同

### 上屏文案

- 官方 API：直接调模型服务
- Web Bridge：控制网页会话并转译结果
- 协议稳定性和能力边界都不同

### 配图建议

双列对比表：

- 左边：官方 API
- 右边：Web Bridge

### 讲稿

这里我想专门花一点时间校正大家的预期。

这套实现和真实 API 调用的最大区别在于，官方 API 是直接向模型服务发请求，输入输出契约通常是显式的、稳定的；而我们这里做的是控制一个网页会话，然后从网页行为里把结果转译出来。

所以从工程上讲，两者的复杂度来源完全不一样。官方 API 更像服务集成；这套实现更像浏览器自动化加协议重建。

另外，tool call、thinking、finish reason 这些东西，在标准 API 里往往是原生协议的一部分；在这里，则需要我们在 Bridge 和 Runtime 里自己重建、补齐、标准化。

所以它的目标不是复刻官方 API，而是尽可能把 Web 端现有能力稳定地转成一个 provider。

## 第 11 页

### 标题

这套实现的限制和风险

### 上屏文案

- 不能 resume 到之前的真实对话
- 目前只支持文字对话
- 并发、稳定性、可恢复性都弱于标准 API

### 配图建议

五个风险图标或一张简化雷达图：

- 页面改版
- 会话绑定
- 自动化失步
- 并发受限
- 延迟波动

### 讲稿

既然它不是标准 API 接入，就一定要正视它的限制。

第一，它目前不能真正 resume 到之前的真实对话。当前实现里虽然维护了一个本地的 `conversationId`，也有 `sessionInit` 和指纹来控制初始化，但本质上还是围绕当前绑定 tab 和当前页面状态在工作，并没有做到按历史会话 ID 恢复远端对话上下文。所以一旦切 tab、重绑、或者页面状态丢失，连续性就会变弱。

第二，它目前只支持文字对话。虽然 runtime 里已经有 image content 类型，但这条链路注册的模型输入仍然是 `text`，图片输入会被降级成占位文本，而不是作为真正的多模态输入发到网页侧。

第三，它在并发、稳定性和恢复能力上，通常不如标准 API。当前 helper 一次只允许一个 active request，请求过程依赖浏览器状态、tab 状态和自动化流程，错误面也更复杂。除了模型本身的响应问题，还会出现 `NOT_BOUND`、`MODEL_BUSY`、`TIMEOUT`、`AUTOMATION_DESYNC` 这类桥接系统特有的问题。

所以这套方案是有明确适用边界的。它的价值很清楚，但不能把它误认为一个和官方 API 完全等价的替代品。

### 讲稿

最后总结一下。

这个项目的价值，在于它把原本只存在于网页里的模型能力，转成了一个本地可调用、可复用、可接入上层 runtime 的 provider。这样系统集成方式就统一了。

但它的边界也很清楚。因为底层依赖的是网页，所以它的稳定性、并发能力、维护成本，和标准 API 方案不是一个量级。

所以这套架构后续最值得投入的方向，不是把它包装得更像官方 API，而是提升这层 Bridge 自己的工程质量，比如更好的可观测性、更强的抗页面变更能力、更清晰的错误恢复机制。

如果用一句话收尾，我会说：这个项目做的，不是封装官方 API，而是把 Web 端模型能力尽可能稳定地转成一个本地 provider。

## 第 10-11 页过渡话术

讲到这里，架构本身已经比较清楚了。但这个项目有一个很重要的前提要说清楚：我们做的不是标准 API 接入，而是一层 Web bridge，所以它的能力边界和风险模型跟真实 API 是不一样的。

## 结尾话术

如果用一句话总结，这个项目做的是：把网页端模型能力，尽可能稳定地转成一个本地 provider。它的价值在于可接入、可统一、可复用；它的限制在于依赖网页、稳定性更脆、维护成本更高。所以后续优化方向，不是把它伪装成官方 API，而是持续提升这层桥接的稳定性和可观测性。

## Q&A 备用回答

### 如果有人问：为什么不直接用官方 API？

如果能直接用官方 API，那会是更标准、更稳定的方案。这个项目的意义不在于替代标准 API，而在于当能力入口是 Web 时，怎么把它工程化地接成一个可复用 provider。它解决的是另一类接入问题。

### 如果有人问：为什么不把所有逻辑都放进 extension？

因为 provider 语义和浏览器自动化语义不是一回事。把它们揉在一起，状态、错误和调试复杂度都会快速膨胀。单独拆出 Helper 和 Bridge，本质上是在控制系统复杂度。

### 如果有人问：这套方案最大的风险是什么？

最大的风险是对网页结构和网页事件行为的依赖。也就是说，这套系统的稳定性不只取决于我们自己的代码，还取决于网页侧是否发生变化。

## 可直接补充到缺陷页的候选点

下面这些点都可以作为“当前项目缺陷”来讲，不一定全部上屏，建议选 4-6 条。

### 第一优先级

- 不能 resume 到之前的真实对话，只能基于当前绑定 tab 延续上下文
- 只支持文字对话，图片输入目前不会作为真正的多模态能力透传
- 一次只能跑一个请求，并发能力弱
- 强依赖页面结构和网页事件，网页改版会直接影响可用性

### 第二优先级

- 会话状态主要保存在本地运行时，重置、重绑或页面丢状态后恢复能力有限
- 调试信息只保留最近一次请求，问题排查深度还不够
- 参数控制能力有限，和标准 API 不能完全对齐
- tool call 需要 bridge 检测和 fallback 修复，稳定性不等于原生 API 协议

## 缺陷页推荐说法

如果你想把“缺陷”讲得更像工程边界，而不是纯负面，可以直接这样说：

当前项目的几个明确缺陷是：

1. 还不能按真实会话维度恢复历史对话，连续性依赖当前绑定页面
2. 目前只支持文字对话，多模态输入链路还没打通
3. 并发能力有限，当前是单请求串行模型
4. 对页面结构和页面协议有依赖，稳定性受外部改版影响
5. 错误恢复和调试能力还比较初级，离生产级 provider 还有距离
